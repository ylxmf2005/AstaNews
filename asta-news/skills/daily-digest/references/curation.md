# 策展细则：评分 rubric 与 prompt 模板

## 四维评分锚点（每维 0-5）

**novelty 信息增量**——"这件事里有多少是世界昨天不知道的？"
- 5：全新能力/方法/事实首次公开（新架构、新训练范式、未见过的能力阈值被跨越）
- 3：已知方向上的实质新进展（明显更好的数字、新的开源权重、协议新版本）
- 1：增量琐碎（小版本修补、已报道事件的跟进碎片、老结果换皮）
- 0：无新信息（营销复述、综述旧闻）
- ⚠️ 注意：novelty 看**信息**新不新，不看**发布日期**新不新。今天发的陈词滥调是 0 分。

**leading_edge 领先性**——"它在推动前沿吗？"
- 5：刷新 SOTA / 行业第一家 / 定义新类别（首个开源 XX、首次在生产环境做到 XX）
- 3：进入第一梯队、紧逼最优（榜单前三的变动、主流框架吸收前沿做法）
- 1：跟随式复现
- 0：与前沿无关

**impact 影响面**——"多少从业者的工作会因此改变？"
- 5：全行业要重估的事（基础模型大版本、核心 infra 范式、重大安全事件）
- 3：一个 stack layer 的从业者需要知道（serving 框架大版本、重要评测变动、新数据配方）
- 1：niche 群体感兴趣
- 0：看完无可行动

**cross_stack 跨层性**——牵动 2+ 层加分项：0（单层）/ 3（两层）/ 5（三层以上，如"新模型+新 serving 优化+新评测"的发布）

加权总分 = Σ rules.scoring.weights × 维度分。**3.0 以上才有资格入选**；2.5-3.0 进雷达。

## 富化 subagent prompt 模板

```
你是 AstaNews 的 {layer_group} 板块研究员。今天是 {date}。
以下是候选条目（JSON）：{candidates}

读者画像与负向兴趣：
{profile.md 内容}

任务：
1. 粗筛：与 AI 全栈无关、命中负向兴趣、纯营销的直接丢弃（不用打分）。
2. 对幸存的每一条【单独】评分：novelty / leading_edge / impact / cross_stack 各 0-5，
   锚点定义：{上方 rubric}。
   拿不准分数时 WebFetch 原文链接核实；仍拿不准 → recommend=false 并写明原因。
3. 每条写一句事实依据（fact）：必须含至少一个量化点（分数、价格、参数量、星标增速…），
   只许用原文已述内容，原文没有就写"原文未给出量化数据"。禁止推断和外推。
4. github release 类条目：读 release notes 判断是否有实质功能（性能数字、新模型支持、
   breaking change），纯 bugfix 版本 recommend=false。

只返回 JSON 数组，每元素：
{"id": "...", "drop": false, "scores": {"novelty": 0, "leading_edge": 0, "impact": 0, "cross_stack": 0},
 "fact": "...", "recommend": true, "primary_url": "若发现更权威的一手源链接则填"}
数组元素必须与你实际评估的条目一一对应，不得多出输入中不存在的 id。
```

（最后一句是防致幻检查：editor 收到结果后核对返回的 id 都在输入集合内，多出的丢弃。）

## Editor 裁决准则

- **3 条优质 > 7 条平庸。** 凑数是大忌。
- **一手源优先**：同一事件，官方 blog/release notes > HF/HN 讨论 > 媒体转述。社区讨论链接放进条目附注（"HN 讨论 N 评论"），不占条目名额。
- **防重**：对照最近 3 天 archive 的标题——同一事件的后续碎片不再报，除非有实质新进展（达到 novelty ≥ 3 的标准）。
- **新与领先压过热度**：HN 800 点的怀旧贴不如 0 讨论的 SOTA 刷新。热度信号（upvotes/points/stars）只用来在同分候选间排序。
- **层数兜底顺序**：若凑不齐 min_layers 层，先从雷达候选里找其他层的 2.5+ 分条目顶上；还不行就减少条数、保住覆盖面声明"今日从简"。
- **数据缺口要诚实**：抓取失败的源、被跳过的 needs_proxy 源，写进 digest 尾部。读者需要知道"没看到 X 是因为没抓到，不是没发生"。
